# overview-flexEMS.svg logo-fix tooling

## What this is for

`overview-flexEMS.svg` embeds a handful of third-party integration logos
(Home Assistant, ENTSO-E, openADR, OpenWeather, weatherapi, and the S2
standard's icon) as `<image>` elements pointing at draw.io's own model.

Two problems this tooling solves:

1. **CSP hardening.** GitHub serves raw SVG files with a `Content-Security-Policy: default-src 'none'` header. Browsers enforce that when the SVG is opened directly (not when embedded via `<img>` in a README, which is unaffected) -- it blocks every `data:` URI image inside the SVG, including these 6 logos. The fix is to inline each logo as literal `<svg>`/`<path>` vector markup instead of a raster `data:` URI, which isn't a "resource load" and can't be blocked by CSP.
2. **Dark-mode support.** Several of these logos need different colors/assets in light vs. dark mode (e.g. ENTSO-E's wordmark turns white, openADR turns fully white, Home Assistant doesn't change, etc.), using the same `light-dark()` CSS / display-toggle `@media (prefers-color-scheme: dark)` mechanism used elsewhere in this diagram.

**The catch:** draw.io fully regenerates the SVG from its own internal
diagram model every time you open and save the file in the app. None of
the above (vector logos, dark-mode CSS) is stored in that model -- only in
the previously-exported SVG text -- so **every time you edit this diagram
in draw.io and save, all 6 logos revert to their original raster/plain
state and lose their dark-mode styling.** This is expected, not a bug in
the fix. Just re-run `fix_logos.py` afterwards.

## Prerequisites

- Python 3, no packages beyond the standard library. `fix_logos.py` only
  splices pre-baked SVG fragments (in `logo_assets/`) back into the file --
  it does not re-derive them from source images, so it needs no
  vectorization library (no `vtracer`, no `potrace`) for normal use.
- You only need `vtracer` (`pip install vtracer`) if you're
  *regenerating* a logo fragment from scratch (e.g. a brand refreshed
  their logo, or you're adding a new integration) -- see "Regenerating a
  logo fragment" below. This is rare.

## Usage

After editing the diagram in draw.io and exporting/saving as
`overview-flexEMS.svg`:

```bash
python3 architecture/scripts/fix_logos.py architecture/overview-flexEMS.svg
```

This edits the file in place. Then, as always:

1. Render it (e.g. open in a browser, or see the repo's usual review process) and check both light and dark mode (browser dark-mode emulation, or your OS theme).
2. Diff it (`git diff`) to confirm only the 6 logo regions changed.
3. Commit.

The script is idempotent-ish: logos it can't find a raster placeholder
for (because they're already patched) are skipped with a `SKIP` message,
not touched twice. Re-running it on an already-fixed file is a no-op.

## How it works

`fix_logos.py` has a small table (`LOGOS`) of the 6 logos, each with:

- a `(width, height)` fingerprint used to locate its raw `<image width="..." height="..."/>` placeholder in the freshly-exported SVG (draw.io keeps these stable even when it shifts `x`/`y` around after a layout edit -- so position is always read fresh from the current file, never assumed)
- the name of its pre-baked replacement fragment in `logo_assets/`
- the `x`/`y` that fragment happened to be baked at, so the script can rewrite it to wherever the logo actually is *now*

Home Assistant additionally gets shrunk to 80% and recentered within its
bounding box every time (its official logomark has no internal padding
and would otherwise overlap the "Home Assistant" caption below it).

`logo_assets/*.svg` are literal SVG fragments (a `<svg x=".." y="..">...</svg>`
or, for the two with dark-mode variants, a `<g>...</g>` pair plus a
`<style>` block toggling between them). They were extracted directly from
a known-good, verified render of `overview-flexEMS.svg` -- they are not
regenerated by this script.

## Regenerating a logo fragment (rare)

If a brand changes their logo, or you're adding CSP-hardening + dark mode
for a new integration icon, you'll need to redo the underlying
vectorization work by hand and re-bake a `logo_assets/*.svg` fragment.
There's no one-command tool for this -- it was done interactively:

1. Source an official vector (SVG) logo if one exists. If not, trace a
   raster with `vtracer` (`pip install vtracer`):
   ```python
   import vtracer
   vtracer.convert_image_to_svg_py(
       "logo.png", "traced.svg",
       colormode="color", mode="spline",
       filter_speckle=4, color_precision=6, layer_difference=16,
       corner_threshold=60, length_threshold=4.0, max_iterations=10,
       splice_threshold=45, path_precision=3,
   )
   ```
2. Strip the full-canvas background rect vtracer emits as its first
   `<path>` (do NOT strip every near-white path -- some of those are
   legitimate letter counters/holes, e.g. openADR's "o"/"p"/"D"/"R").
3. **If the logo has letters with holes (o, p, e, A, D, R, ...):**
   vtracer traces each hole as a *separate*, near-white-colored path
   layered on top of the solid letter -- this only looks like a real hole
   against a non-white test background, and renders as solid/filled
   against this diagram's actual white panels. It is NOT a real
   transparent cutout. Don't try to fix this with an SVG `<mask>` either
   -- masks silently fail to render when the SVG is loaded via
   `<img src="...">` (this repo's actual deployment context, confirmed
   with a minimal reproduction), even though they work fine inline.
   The real fix is to merge each letter's colored path and its
   corresponding hole path into a single compound `<path>` (concatenate
   both sub-path `d` strings, coordinate-adjusted into the same local
   frame) with `fill-rule="evenodd"`. This produces a genuine geometric
   hole that works in any rendering context.
4. For dark-mode variants: either recolor in place with
   `style="fill:light-dark(#light,#dark)"` (simple single-color swaps,
   works fine -- but NOT for elements that also need `fill:none` +
   `stroke:light-dark(...)`; librsvg/GNOME viewers don't reliably parse
   `light-dark()` and a stroke-only path with fill:none can end up with
   neither fill nor stroke, i.e. invisible), or duplicate the whole
   fragment as two `<g id="...">` siblings toggled via `display` and
   `@media (prefers-color-scheme: dark)` (safer, no CSS color functions
   involved, works in browsers and static SVG viewers alike -- this is
   what openADR/OpenWeather/S2/weatherapi use here).
5. Verify by rendering standalone via `<img src="fragment.svg">` in an
   actual browser (headless Chromium via Playwright works well) at full
   resolution -- small/thumbnail renders can hide a still-broken hole.
   Check both light and dark (`colorScheme` in Playwright, or your OS
   theme).
6. Save the final fragment into `logo_assets/`, and add/update its entry
   in the `LOGOS` table in `fix_logos.py` (fingerprint width/height, baked
   x/y).
